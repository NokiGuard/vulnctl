# Releasing vulnctl

How to cut, sign, publish, and verify a vulnctl release. The pipeline lives in
[`.github/workflows/release.yml`](../.github/workflows/release.yml).

## What the pipeline does (and why)

Four jobs, each gating the next. Nothing reaches PyPI until every gate and the
signing step have passed.

| Stage | Does | Security rationale |
|---|---|---|
| **build** | `uv build` → sdist+wheel; install the wheel into a clean venv; SBOM that closure (Syft, CycloneDX); scan it (Grype); **Gate 1** blocks any *critical-with-fix* vuln; **Gate 2** dogfoods `vulnctl … --fail-on act` on its own Grype report | The SBOM/scan describe the *exact* closure a `pipx install vulnctl` user gets. Gate 2 runs the wheel we're about to ship, so a KEV-listed dependency (→ `exploitation=active` → ACT) blocks the release. A vulnerable artifact never gets signed or published. |
| **sign** | cosign **keyless** (OIDC) signs the wheel, sdist, and SBOM → `.sig` + `.pem` per file | No long-lived signing key exists to leak. Fulcio issues a short-lived cert bound to *this workflow's* identity; the signature is logged in the public Rekor transparency log. |
| **publish** | PyPI **trusted publishing** (OIDC) uploads the sdist+wheel; PEP 740 attestations generated by the same identity | No PyPI API token is stored anywhere. PyPI verifies the OIDC identity directly. |
| **release** | GitHub Release with wheel, sdist, SBOM, and signatures attached | Consumers can fetch and verify artifacts + provenance from one place. |

**Least privilege:** the top of the workflow sets `permissions: {}` (deny all);
each job grants only what it needs. `id-token: write` (OIDC) is present **only**
on `sign` and `publish`. `contents: write` is present **only** on `release`.
Every `uses:` is pinned to a full commit SHA.

## One-time setup

1. **PyPI trusted publisher.** On <https://pypi.org> → project `vulnctl` →
   *Publishing* → add a GitHub Actions trusted publisher:
   - Owner `NokiGuard`, repository `vulnctl`
   - Workflow filename `release.yml`
   - Environment name `pypi`

   (For the very first upload, register a *pending* publisher before the project
   exists, or do the initial upload another way.)
2. **GitHub Environment.** Repo *Settings → Environments →* create `pypi`.
   Optionally add required reviewers so a human approves before publish.
3. *(Optional)* repo secret `VULNCTL_NVD_API_KEY` to raise NVD rate limits during
   Gate 2. vulnctl fails open without it.

## Dry run (no publish)

Every part of the pipeline *except* the PyPI upload and the GitHub Release runs
on a manual dispatch — including SBOM, scan, both gates, and real keyless
signing — so you can validate the whole path before tagging:

1. GitHub → **Actions → release → Run workflow** (`workflow_dispatch`).
2. Confirm all steps are green. Open the run's **Artifacts**: `dist` (sdist+wheel),
   `reports` (`sbom.cdx.json`, `grype.json`), `signatures` (`.sig`/`.pem`).
3. Optionally verify a downloaded signature with the command below.

> A dry run performs real keyless signing, which writes a public **Rekor
> transparency-log entry**. That's expected and harmless — it does not publish
> the package.

## Cut a release

1. Ensure `main` is green (CI passing).
2. **Update the changelog:** move `## [Unreleased]` items into a new
   `## [X.Y.Z] - <date>` section in [`CHANGELOG.md`](../CHANGELOG.md).
3. **Bump the version** (single source of truth) in
   [`pyproject.toml`](../pyproject.toml) → `[project].version`.
4. Commit and merge to `main` (`chore: release vX.Y.Z`).
5. **Dry run** (above) from the merge commit; confirm green.
6. **Tag and push** — the tag must match the `pyproject` version (the build job
   enforces this):

   ```bash
   git tag -a v0.1.0 -m "vulnctl v0.1.0"
   git push origin v0.1.0
   ```

7. Watch the `release` run. On success: PyPI has the release, and the GitHub
   Release carries the wheel, sdist, SBOM, and signatures.

## Verify signatures

Download the release assets, then verify each blob against the workflow's
signing identity (keyless — no public key file):

```bash
VER=0.1.0
IDENTITY='^https://github.com/NokiGuard/vulnctl/\.github/workflows/release\.yml@refs/tags/v.*$'
ISSUER='https://token.actions.githubusercontent.com'

for f in "vulnctl-$VER-py3-none-any.whl" "vulnctl-$VER.tar.gz" sbom.cdx.json; do
  cosign verify-blob \
    --certificate "$f.pem" \
    --signature   "$f.sig" \
    --certificate-identity-regexp "$IDENTITY" \
    --certificate-oidc-issuer "$ISSUER" \
    "$f"
done
```

Each file should report `Verified OK`. A mismatch means the artifact wasn't
produced by this repo's release workflow — do not trust it.

Then verify the install itself:

```bash
pipx install "vulnctl==$VER"
vulnctl --version           # → vulnctl 0.1.0
```

PyPI also shows the PEP 740 attestations on the project's release files page.

## Yank a bad release

PyPI releases are **immutable** — you cannot re-upload the same version. If a
release is broken:

1. **Yank on PyPI:** project → *Manage → Releases →* the version → *Options →
   **Yank***. Yanking keeps it installable only by exact pin and hides it from
   normal resolution. No token needed (web UI).
2. **Fix forward:** bump to the next patch (e.g. `0.1.1`), update the changelog,
   and run this process again.
3. *(Optional)* delete or mark the GitHub Release/tag if its assets are wrong.

Never delete a PyPI release to "replace" it — that version number is burned
forever; always ship a new one.
